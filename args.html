## dig deeper ```cpp // .hpp void foo(T); // .cpp void foo(T const); ``` ```cpp void bar(S); S foo(); bar(foo()); // copy elision, zero cost S s; bar(s); // copy, can be expensive ``` ```cpp void foo(S*); void foo(std::optional); // by value and optional, kinda immutable auto p = std::optional(nullptr); if(! p) // flase, p has a value if(p == nullptr) // true, the value is nullptr ``` `std::smart_ptr&` can be dangerous >R.37: Do not pass a pointer or reference obtained from an aliased smart pointer ```cpp auto gp = std::make_shared(); void foo(T*); void bar(std::shared_ptr& p) { // some code foo(p.get()); } bar(gp); ``` ```cpp void bar1(std::shared_ptr p) { foo(p.get()); } void bar2(std::shared_ptr& p) { auto c = p; foo(c.get()); } // more cumbersome to std::unique_ptr ``` ## Aliasing ![code](https://godbolt.org/z/16Tcv6) That's why most xxx_cast you see is undefined behavior - pass by value when you can, pass by const ref when you must - use raw pointer for optional, unique_ptr/shared_ptr for lifetime - don't share with \* or &, express via shared_ptr ## Move semantics solves these and *may* provide a performance boost ```cpp struct S { int a[1000]; }; // no owned resources struct S { std::string s; }; // with SSO, the performance is no better than copy ```
